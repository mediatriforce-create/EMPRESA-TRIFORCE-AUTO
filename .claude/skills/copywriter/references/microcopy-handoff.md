## Princípios Fundamentais de Microcopy

Microcopy é a copy invisível que determina se o usuário avança ou abandona. Cada elemento tem regras específicas. Clareza > criatividade em todo contexto de interface.

- **Clareza > criatividade** — o usuário não deve parar para pensar
- **Brevidade com propósito** — cortar até o ponto em que cortar mais perde significado
- **Linguagem do usuário** — zero jargão técnico
- **Consistência** — mesma ação = mesmas palavras em todo o sistema
- **Voice constante + tone adaptável** — personalidade fixa, tom muda por contexto

## Guia por Tipo de Elemento

### Botões / CTAs

**Estrutura:** verbo de ação + resultado específico

**1ª pessoa converte mais:**
- "Quero meu desconto" > "Obter desconto"
- "Quero agendar meu horário" > "Agendar horário"
- "Garantir minha vaga" > "Cadastrar"

**Pares claros obrigatórios:**
- Salvar / Cancelar
- Excluir / Manter
- Enviar / Salvar rascunho
- Confirmar / Voltar

**Aprovado:**
- "Salvar alterações"
- "Agendar agora"
- "Ver disponibilidade"
- "Enviar mensagem"
- "Quero minha LP"
- "Falar no WhatsApp"

**Reprovado:**
- "OK" (ambíguo)
- "Enviar" (sem contexto)
- "Clique aqui" (sem ação)
- "Saiba mais" (sem contexto — onde? o quê?)
- "Sim" / "Não" (em botões isolados)

**Limite:** 20-30 chars para botões primários | 30-40 chars para CTAs contextuais

---

### Labels de Formulário

**Regras:**
- Sempre visíveis (não usar placeholder como substituto de label)
- Sentence case: "Nome completo" > "NOME COMPLETO"
- Concisas: "E-mail" > "Digite seu endereço de e-mail aqui"
- Obrigatório indicado discretamente: asterisco (*) + legenda no topo do formulário: "* campo obrigatório"

**Aprovado:**
- "Nome completo"
- "WhatsApp" (não "Número de telefone")
- "Qual serviço você quer agendar?"
- "Mensagem (opcional)"

**Reprovado:**
- "NOME:" (caixa alta + dois pontos)
- "Por favor, digite aqui o seu nome completo"
- "Telefone/Celular/WhatsApp" (ambíguo)

---

### Placeholders

**Papel:** suporte/exemplo, NUNCA substituto de label.

**Aprovado:**
- "ex: joao@gmail.com" — mostra formato esperado
- "ex: (11) 99999-9999" — mostra formato de número
- "ex: Corte + barba"

**Reprovado:**
- "Digite seu e-mail" — duplica a label, some ao clicar, usuário perde referência
- "Escreva aqui" — sem informação adicional
- "Campo obrigatório" — isso é da label, não do placeholder

**Limite:** 30-40 chars

---

### Mensagens de Erro

**Estrutura 3 partes obrigatória:**
1. O que aconteceu
2. Por que aconteceu
3. Como corrigir

**Aprovado:**
- "E-mail inválido. Verifique se digitou corretamente, como nome@exemplo.com"
- "Número de telefone incompleto. Digite com DDD, ex: (11) 99999-9999"
- "Campo obrigatório. Por favor, preencha seu nome para continuar."

**Reprovado:**
- "Erro de validação"
- "Campo inválido"
- "Tente novamente"
- "Você errou..." (culpa o usuário)

**Tom:** nunca culpar o usuário.
- "Não foi possível enviar" > "Você não enviou"
- "Este e-mail não foi encontrado" > "Você errou o e-mail"

**Limite:** 80-120 chars

---

### Empty States

**3 componentes obrigatórios:**
1. O que pertence aqui
2. Por que está vazio
3. Call to action para preencher

**Aprovado:**
- "Seus agendamentos aparecerão aqui. Marque seu primeiro horário."
- "Nenhuma mensagem ainda. Quando entrar em contato, a conversa aparece aqui."
- "Sua lista de favoritos está vazia. Explore nossos serviços."

**Reprovado:**
- "Nenhum item encontrado"
- "Lista vazia"
- "404 Not Found" (em empty state de aplicação, não em erro de rota)

---

### Onboarding e Progresso

**Mostrar valor antes de pedir esforço:**
- ❌ "Primeiro, preencha seus dados pessoais"
- ✅ "Em 3 passos simples você agenda seu horário"

**Progresso explícito:**
- "Etapa 2 de 4 — quase lá"
- "Falta apenas 1 passo"

**Encorajamento progressivo:**
- "Ótimo! Agora falta só o horário."
- "Quase lá — confirme seu agendamento."

---

### Success States

**Confirmar o que aconteceu + o que acontece agora:**

**Aprovado:**
- "Agendamento confirmado! Você receberá uma confirmação por WhatsApp em instantes."
- "Formulário enviado. Nossa equipe entrará em contato em até 2 horas."
- "Dados salvos com sucesso. Você pode fechar esta janela."

**Reprovado:**
- "Sucesso!"
- "Formulário enviado."
- "OK"

## Voice Check — Framework de Revisão de Microcopy

5 perguntas antes de aprovar qualquer microcopy:

1. **Voice** — tom consistente com `brand/voice.md`? (direto, sem enrolação, confiante mas acessível)
2. **Issues** — quais elementos têm problemas de clareza, ambiguidade ou tom errado?
3. **Patterns** — erros recorrentes que indicam problema sistêmico de voz ou estrutura?
4. **Quick Wins** — mudanças de 5 minutos com alto impacto de clareza ou conversão?
5. **Zero friction** — o usuário precisa pensar para entender qualquer elemento?

---

## 5 Testes de Validação de Microcopy

1. **Readback test** — leia em voz alta. Se soar estranho, reescreva.
2. **Screenshot test** — sem contexto, a mensagem é clara? Teste com alguém que não conhece o produto.
3. **Stress test** — e se o texto tiver o dobro do tamanho? (nome de 40 chars no campo de 20)
4. **Translation test** — funciona em outro idioma? Se não, pode ser idiomático demais.
5. **Truncation test** — como fica cortado em mobile 375px? Sentido preservado?

## Template de Entrega de Microcopy para Felipe

Este formato é obrigatório em toda entrega de copy. Felipe consome diretamente para implementação.

```markdown
# [Nome do Projeto] — Copy Doc
Versão: X.X | Data: YYYY-MM-DD | Copywriter: Fecchio

---

## [Nome da Tela / Componente]

### Copy Principal
- **Headline:** [texto — máx X chars]
- **Subtítulo:** [texto — máx X chars]
- **Corpo:** [parágrafos]
- **CTA Primário:** [texto do botão — máx 30 chars]
- **CTA Secundário:** [texto — máx 40 chars]
- **Meta title:** [50-60 chars]
- **Meta description:** [150-160 chars]

### Microcopy
| Elemento | ID do componente | Copy | Estado | Máx chars |
|---------|-----------------|------|--------|-----------|
| Botão CTA | btn-cta-hero | Quero agendar agora | default | 30 |
| Botão CTA | btn-cta-hero | Aguarde... | loading | 20 |
| Input nome | input-nome | — | default label | — |
| Input nome | input-nome | ex: João Silva | placeholder | 40 |
| Input nome | input-nome | Nome obrigatório. Digite seu nome completo. | error | 80 |
| Input email | input-email | — | default label | — |
| Input email | input-email | ex: joao@gmail.com | placeholder | 30 |
| Input email | input-email | E-mail inválido. Verifique o formato, como nome@exemplo.com | error | 80 |
| Form success | form-success | Agendamento confirmado! Você receberá uma confirmação por WhatsApp. | success | 100 |

### Notas de implementação (para Felipe)
- Chave i18n sugerida: `[tela].[componente].[elemento]`
- Erro aparece inline sob o campo, não em toast
- aria-label obrigatório no botão de agendamento
- Botão WhatsApp: URL pré-preenchida com mensagem do campo de serviço
- Copy condicional: se usuário já agendou antes → "Agendar novamente" em vez de "Quero agendar agora"

### Notas de design (para Camila)
- CTA primary: máx 24 chars para não quebrar em mobile 375px
- Mensagem de erro: espaço suficiente para 2 linhas sob o campo
- Empty state: ícone + headline + CTA (não só texto)
- Tom da seção: [específico por seção]
```

## Convenção de Nomenclatura de Elementos

**Padrão:** `[Tipo][Funcionalidade][Estado]`

**Exemplos:**
- `ButtonCTA` / `ButtonCTALoading` / `ButtonCTADisabled`
- `InputEmail` / `InputEmailError`
- `CardServiço` / `CardServiçoSelected`
- `BadgePromo` / `BadgeAvaliaçãoGoogle`
- `FormAgendamento` / `FormAgendamentoSuccess`

**Chaves de copy:** `copy.[tela].[componente].[elemento]`
- `copy.home.hero.title`
- `copy.home.cta.label`
- `copy.form.email.error`
- `copy.form.success.message`

## Ferramentas de Handoff no Stack Triforce Auto

| Ferramenta | Quando usar | Quem consome |
|-----------|-------------|-------------|
| **Arquivo Markdown no repo** | `/src/copy/[tela].md` — principal, versionado com código | Felipe (implementação) + Camila (referência de copy) |
| **Figma comments** | Anotações de copy direto no componente | Camila anota via Fecchio input, Felipe consome |
| **Strings file opcional** | `/src/i18n/pt-BR.json` — chaves de copy reutilizável | Felipe para componentes globais |

**Boas práticas de handoff:**
- Uma seção por tela/componente — não misturar telas no mesmo bloco
- Indicar limite de caracteres para elementos críticos (botões, headlines mobile)
- Sinalizar condicionais: "se usuário já agendou antes → copy alternativa"
- Entregar em Markdown (legível no GitHub, Notion, VSCode e Figma comments)
- Sempre incluir: copy padrão + copy de estado vazio + copy de erro + copy de sucesso
